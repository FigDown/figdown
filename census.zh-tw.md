# 圖型普查 — 真實語料（2026-07-02）

> FigDown 首版範圍（R12：量大內容優先）與預設值（R13：預設 = 最常見
> 情況）的數據依據。
>
> English version: [census.md](census.md)

## 語料與方法

- 來源：**774 份內部硬體設計文件**（網路交換器領域，.docx）的私有
  語料。文件內容不公開；此處只出現彙總統計。
- 抽取：自 docx 容器抽出全部內嵌圖片——**12,342 張**（53% EMF 向量、
  45% PNG、2% WMF/JPEG）。
- 以 MD5 去重 → **2,177 張唯一圖片**（82% 的出現次數是重複：共用
  模板、跨文件複用的圖）。
- 分類：每張唯一圖片由 AI 視覺辨識逐張分類，分類法取 **Mermaid、
  PlantUML、D2、Graphviz、WaveDrom 支援圖型的聯集**，另加非圖形桶
  （截圖/照片/其他）。覆蓋率 2,127/2,177（97.7%；其餘 ~2.3% 待
  限額窗口補跑，不影響排名）。後續將由人工校正分類資料夾。

## 對 FigDown 的結論

1. **v0.1 範圍獲得證實並量化。** 核心場景（方塊架構 + 拓撲 + 流程圖，
   同一個「方塊+連線」模型）= 加權 37.5%；加上三個型別區塊——
   `bitfield`（23.7%）、`table`（9.8%）、`wave`（7.2%）——覆蓋
   **全部非模板圖出現次數的 78%，約等於可分類圖形的 94%**。
2. **優先順序**（加權）：核心場景 → bitfield → table → wave。兩大
   巨頭是方塊架構圖與封包/位元欄位圖，合計約 48%——恰好都是 Mermaid
   做不好或做不了的類型。
3. **Mermaid 覆蓋率假設被修正。** Mermaid 原生穩定支援的圖型
   （flowchart、sequence、state、class-ER、圖表、mindmap）在本語料
   合計只有 **約 12%**——遠低於最初「約一半」的估計（R0）。在硬體
   文件中，FigDown 瞄準的缺口是絕對多數，不是剩餘部分。
4. **向量原稿佔多數**（53% EMF）：大部分圖是被「畫」出來的，不是
   拍攝或截圖——它們生而有結構、只是被容器丟掉了。文字格式是在
   **找回**本來存在的東西，不是發明新東西。（直接解析 EMF 作為
   遷移輔助是一個待議想法。）
5. `other`（唯一圖的 15.3%）多為複合圖的碎片、裝飾元素與不可讀的
   殘片；人工校正會把其中一部分重新歸位。
