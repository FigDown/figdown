# FigDown 標準 — 需求討論筆記

> 專案名：**FigDown** — *Figures as text, one source, two readers.*（見 D2）
>
> English version: [requirements-notes.md](requirements-notes.md)

狀態：需求收集中（2026-07-01 起）。
目的：把先行編輯器原型的做法推廣成正式標準（文字為源、圖為產物、
雙讀者共享知識；下文以 v1/v2 指該原型的兩版草案）；本檔逐條記錄使用者口述的需求，整理定稿後再轉寫為
正式標準文件。

（記錄慣例：使用者以口語闡述，「原意」段落經過文句修飾——忠於語意，
不逐字保留口語措辭；「整理後解讀」是記錄者的分析與推論。）

## 背景（既有實作的核心要素，供對照）

1. 文字是唯一真相 — GUI 每個操作 = 代使用者編輯文字
2. 語言先行（nudge 先例）— 先加語言指令、後做 GUI
3. 引擎不 fork — editor 由 build script 自引擎單檔抽取組裝
4. 封閉文法紅利 — 補全可窮舉、錯誤帶行號、錯誤恢復容易
5. 雙向同步 — 游標 ↔ 預覽步驟；步驟側欄 = 文字段落的導覽投影

### R0 — 原始需求來源（2026-07-01）

**情境**：手上有大量 Word 文件，內含大量圖片（很常見的場景）。圖片種類繁多：

- 約一半可用 Mermaid 這類語法描述（如流程圖）
- 另外很多不行（如網路拓撲等）

**動機**：希望把這些文件中的知識**與 AI agent 共享**。圖片本身 AI 難以
看懂（知識被鎖在點陣圖裡）。

**目前判斷**：轉成 Mermaid 看起來是最合適的方向（但保持開放：也許有
其他同樣好或更合適的方案）。問題是很多圖片無法直接轉出 Mermaid。

**期望方向**：**擴充 Mermaid**（或等效的文字化方案），同時必須兼顧
**人類獲取知識的習慣與方式**——不能只是 AI 可讀、人類閱讀體驗倒退。

**關鍵洞察**：圖片中的 **layout（版面配置）對理解很有幫助**——文字化
方案不能丟掉排版所承載的語意。

**整理後解讀**：這是一個「圖片知識文字化」的雙受眾問題——
同一份表示法要讓 (1) AI 能讀懂並運用其中知識、(2) 人類保有接近原圖的
理解體驗（含 layout 語意）。Mermaid 是基準參考點，缺口在於它覆蓋不了
的圖型（拓撲類等）與 layout 表達力。

（後續：使用者將逐點描述期望 → R1, R2, …）

### R1 — 靜態優先，動態後期支援（2026-07-01）

**原意**：主要至少要支援**靜態**圖；後期最好能支援**動態**——動態
= 靜態 + 「換 page」的概念（例如 GIF 那樣一幀一幀）。

**整理後解讀**：
- 優先級明確：靜態是 must-have 基線，動態是 nice-to-have 的後期目標。
- 動態的心智模型是**離散翻頁**（page/frame），不是連續時間軸動畫——
  與 v2 草案的定調完全一致（「動態 = 靜態 + step timeline」、
  步驟模型與 頁/張/幀 天然同構、匯出 GIF = 步間取幀）。
- 這意味著標準的分層應該是：靜態場景層先定，動態只是在其上疊
  step/page 序列，而非兩套語言。

### R2 — 動態的用途定位（2026-07-01）

**原意**：動態的用途，是像原型編輯器的範例那樣：把演算法、
或含步驟/流程的內容做成動態呈現，用於**展示與說明**——這類內容以
動態圖表達會比靜態圖更容易理解。

**整理後解讀**：
- 動態的定位是**教學/講解工具**（展示與說明），不是文件知識庫的
  必要成分。對象是演算法、協議步驟、流程——本質上有時間序的內容。
- 換句話說，兩個受眾場景開始分化：
  - **靜態**：服務 R0 的核心需求（Word 文件知識讓 AI+人類共享）
  - **動態**：服務展示/說明場景（= 現有 v2 原型已在做的事）
- 現有原型被使用者視為動態場景的正面範例，標準應把其做法收編
  為動態層的參考，而不是重造。

### R3 — 文字為源、圖為產物（單一真相，雙讀者投影）（2026-07-01）

**原意**：人類最終可能以「圖」或「多張圖」（動畫/動態）來理解；但期望
其中相同的「內容」與知識價值也能讓 AI 清楚理解。所以主要期望是**類似
Markdown：文字描述為主（for AI），但能「標準化」轉成「圖或多張圖」呈現
給人類**。這來自兩種不同「讀者」群的不同需求——但同時**不想維護兩套
幾乎同義的內容**。結論：**文字為主（來源），圖片為輔（產物）**的方式
維護。

**整理後解讀**：
- 這條確立了整個標準的**架構公理**：
  1. **單一真相 = 文字**（MD 式、行文可讀，AI 直接消化）
  2. **圖 = 確定性的渲染產物**（人類讀者的投影），一份靜態圖、
     多張圖 = 動態（呼應 R1 的換頁模型）
  3. **嚴禁雙源**：不允許「文字一份 + 圖一份」各自維護——圖必須
     100% 由文字生成，改文字即改圖
- 「標準化轉成圖」意味著轉換必須**確定性**（同文字 → 同圖），否則
  圖無法被信任為文字的忠實投影——與 v2 草案的確定性原則同源。
- 這正是原型編輯器鐵律「文字是唯一真相；GUI = 代使用者編輯
  文字」在文件場景的推廣：editor 的 GUI 是投影，這裡的「圖」也是投影。
- 與 Mermaid-in-Markdown 的既有生態同構（```mermaid 圍欄 → 渲染圖），
  使用者要的是把這個模式擴充到 Mermaid 覆蓋不了的圖型（R0）。

### R4 — 文字→圖必須由工具全自動完成（2026-07-01）

**原意**：「文字轉圖片」這個動作，自然期待由**工具**完成——**絕不會
是人類**來轉。

**整理後解讀**：
- 渲染管線必須**全自動、零人工介入**：人（或 AI）只寫文字，圖由
  工具產出。人類永遠不手畫、不手調渲染結果。
- 推論一：語言裡必須內建足夠的 layout 表達力（呼應 R0「layout 對
  理解有幫助」）——因為沒有人工後製這條退路，排版語意必須在文字
  來源裡就說得清楚，工具才排得出「像原圖」的結果。
- 推論二：工具的品質門檻很高——「自動排出來的圖夠不夠好看/夠不夠
  像人畫的」直接決定這個標準能不能活。這正是原型專案累積
  的核心能力（宣告式 layout、三層級排版、確定性）。
- 推論三：escape hatch 的形式也必須是「寫在文字裡的宣告」（如 v1
  的 `pos`/`nudge`），而不是「人工微調產出的圖」。

### R5 — 內容易表達之外，人類要有（optional）layout 控制權（2026-07-01）

**原意**：文字（source content）除了容易表達內容與意義之外，還希望讓
人類有 layout 等控制權。圖片本質上由很多幾何圖形（含線條）和 **layers**
組成，每個物件有**顏色**等屬性——這是很基本的圖片需求。即使是流程圖，
也要有彈性讓人類編輯時有一定控制權：例如**控制某些物件在整張圖的某個
絕對位置**，目的是讓人類能輕易表達出「**位階**」等概念。所以文字上
需要有 **optional** 的彈性讓用戶（人類）操作。

**整理後解讀**：
- 語言有兩個層次的訴求，缺一不可：
  1. **語意層**（主體）：容易表達內容與意義（AI 讀寫的主戰場）
  2. **展現層**（optional 加註）：幾何、線條、layers、顏色、絕對
     位置等控制權，人類按需取用，不寫則全自動
- 關鍵動機：**位置本身承載語意**（位階、分區、方向）。人類要求絕對
  位置控制不是為了美術微調，而是為了表達「這個東西地位比較高/屬於
  某區」——排版即知識的一部分（呼應 R0 的 layout 洞察）。
- 對應 v2 草案已定的**三層級排版**：全自動 → 語意約束（flow/rank/
  zone）→ 明確位置（grid/pos escape hatch）。R5 確認了這三層都要，
  且「不寫 = 全自動」的 optional 性質。
- **與 v1 鐵律的張力（待釐清）**：v1 規定「作者永不選
  顏色」（plane→顏色固定）；R5 卻要求物件有顏色等基本屬性控制。
  可能的調和：文件場景（靜態圖）開放展現屬性、協議動畫 profile
  維持語意配色——需與使用者確認邊界。
- 「layers」（圖層）是新出現的一級概念：v1 只有固定的渲染層堆疊
  （§6.6，引擎內部），R5 要求把 layer 作為作者可用的組織單位。

### R6 — 一定要有編輯器（2026-07-01）

**原意**：一定要能有編輯器，方便人類**繪製**想要的圖。

**整理後解讀**：
- 編輯器是標準的**硬性要求**（must-have），不是配套工具的 nice-to-have。
  人類的創作動線是「畫圖」，不是「寫程式碼」——即使真相是文字，
  人類要有視覺化的操作介面。
- 與 R3/R4 公理合成，得到的必然形態就是原型編輯器的鐵律：
  **GUI 繪製動作 = 代使用者編輯文字**。人類「畫」，編輯器把畫的
  動作轉寫成文字宣告（拖節點→pos、連線→link、調色→屬性行）；
  文字仍是唯一真相，圖仍是投影。
- 這也解答了 R5 的操作面：人類的 layout 控制權主要透過編輯器的
  直接操作行使（拖到哪 = 寫入絕對位置/位階宣告），而不是要求
  人類手寫座標。
- 推論：語言設計必須顧及「可被 GUI 回寫」——每個視覺操作都要有
  對應的文字指令（nudge 先例），這是語言與編輯器的雙向契約。
- 既有資產：原型編輯器的架構（CodeMirror + canvas 直接操作 +
  build-from-engine）可直接作為參考實作。

### R7 — 正式標準、通用性、AI 教學成本要低（token 精簡）（2026-07-01）

**原意**：這套文字格式**必定需要制定標準**，希望能**通用**；也希望
「說明這套語法」的內容能**精簡**——不需要消耗太多 token 來提供 prompt
讓 AI agent 看懂。

**整理後解讀**：三個子需求：
1. **正式標準化**：要有 normative spec（完整參考等級：
   語法、語意、錯誤模型、EBNF），不是「約定俗成的工具格式」。
2. **通用性**：跨場景（不綁死在網路協議）、跨工具（任何人可依 spec
   實作 renderer/editor）、可互換。
3. **AI 教學成本是一級設計指標**：把「讓 AI agent 學會這個語言所需
   的 prompt token 數」當成語言設計的硬約束——語法說明要能塞進
   一個精簡的 system prompt（數百行級，而非整本 spec）。
- 設計含義：
  - 語言核心要小而正交（少數指令 + 一致的組合規則），靠規則性而
    非長清單讓 AI 泛化——v1 的封閉文法+行導向正是這種性質。
  - spec 可以分兩份受眾文件：完整 normative spec（給實作者）+
    精簡 authoring guide（給 AI agent 的 prompt，如 v1 spec §8 的
    獨立抽出版）。
  - 「通用」與「精簡」互相拉扯（涵蓋圖型越多語言越大），標準需要
    profile/分層機制：小核心 + 按需載入的領域擴充，AI 只需被教
    當下用到的 profile。
  - 另一個省 token 的槓桿：語法貼近 AI 已熟悉的慣例（Mermaid/DOT
    等），先驗知識可遷移（usecases.md 場景 1 的 build-vs-buy 已有
    此結論）。

### R8 — 渲染標準：人類指定優先 + 小改動不得大變圖（2026-07-01）

**原意**：「文字→圖片」的過程必須有**更詳細的標準**。元件若有明確
參數（位置、顏色等屬性），**優先以人類指定的為準**；其餘未指定的部分
才由演算法自動調配與適應。主要目的：當原始內容（文字）**修改幅度小**
時，新舊圖的差異**不該過大**（否則人類要重新理解整張圖）。

**整理後解讀**：
- 兩條渲染規範：
  1. **優先權規則**：explicit（人類指定）> 演算法自動。自動排版
     必須「繞著」被釘住的元件調配，不得覆蓋人類意志。
  2. **排版穩定性（layout stability）**：小的輸入差異 → 小的輸出
     差異。這比 v2 已有的「確定性」（同文字→同圖）**更強**：
     確定性是同輸入的可重現，穩定性是**相鄰輸入間的連續性**。
- 動機是認知成本：人類對圖有空間記憶（「FIB 表在右上角」），改一行
  字導致全圖重排 = 摧毀讀者已建立的心智地圖。
- **設計後果（重大）**：
  - usecases.md 記載 partial pinning（自動+釘住少數節點）原本
    「留待日後」——R8 把它升級為**核心需求**，不能再延後。
  - 純 dagre/ELK 式全域重算天生不穩定（加一個節點可能全圖翻面），
    標準需要明訂穩定化手段：釘點約束、增量排版、或「排版結果可
    物化回文字」（把算出的位置寫回成 optional 宣告，凍結現狀）。
  - 「物化回文字」與 R6 編輯器鐵律天然契合：人類在編輯器裡認可的
    佈局＝寫回文字的 pos 宣告，之後的小改動自然穩定。
  - 這條也是對「為什麼不直接用 Mermaid/Graphviz」的最強技術論證
    ——它們都沒有跨版本佈局穩定性的承諾。

### R9 — 產物形式：SVG 嵌入 .md；AI 讀文字、人讀圖（2026-07-02）

**原意**：為了方便使用者閱讀，這個文字轉圖片的標準可以讓 **LLM model
轉出對應的圖**。預設考慮 **SVG 向量圖**——方便嵌入 .md 引用，不用擔心
MD reader/viewer 不支援 mermaid 等。但當 **AI 讀這份文件時，要可以
捨棄「圖」**，直接拿原始「文字」來理解。（使用者自註：很多部分還需要
討論，也許已有更好的方案。）

**整理後解讀**：
- **產物格式定調：SVG 向量圖**，理由是可移植性——嵌入 .md 後任何
  viewer 都能顯示（img 引用/直嵌皆可），不依賴 viewer 端的 mermaid
  渲染支援。這把「渲染時機」從讀取端（Mermaid 模式：viewer 現場渲染）
  移到**寫入端**（產出時渲染、隨文件攜帶）。
- **雙通道閱讀模型**：同一份 .md 中，
  - 人類通道：看嵌入的 SVG
  - AI 通道：跳過 SVG，直接讀原始文字宣告
  - → 標準需明訂兩者的**配對關係**（文字塊與其產物 SVG 如何關聯、
    如何標記「此圖由彼文字生成」、如何偵測過期不同步）。
- **⚠ 與 R4/R8 的張力（待釐清，使用者也自覺未定）**：
  R4 說轉換「絕不由人類」、R8 要求確定性+穩定性；R9 卻提到「讓 LLM
  轉出對應的圖」。LLM 不是確定性工具——同輸入不保證同輸出。
  兩種可能的解讀：
  (a) LLM 只是**呼叫者**：LLM 寫文字、跑確定性 renderer 工具產 SVG
      （相容 R4/R8，較可能是本意）
  (b) LLM 本身是 renderer：直接生成 SVG（違反 R8 穩定性，除非標準
      詳細到把自由度壓光——呼應 R8「更詳細的標準」的說法）
  下次討論需確認。若是 (b)，標準的角色會變成「約束 LLM 輸出的渲染
  規範」，工程形態完全不同。
- 使用者明示這條保持開放：歡迎更好的方案（如既有標準/工具能覆蓋）。
- **→ 張力已解決，見 D1（2026-07-02）**。

### R9.1 / D1 前置 — renderer 澄清：程式優先，LLM 僅過渡（2026-07-02）

**原意**：R4 所說的「工具」，優先希望**不依賴 LLM API** 就能實現。
只要標準的文字語法制定得好，就能像 mermaid 一樣**直接由程式轉換**，
而不必請 AI/LLM agent 按某套規則重新產生。初期可以接受方案 B
（LLM 代轉）作為過渡，但最終目標是方案 **(A)**：無論文字出自人手
或 AI（人也可能透過編輯器生成、調整），都必須**穩定產出 1:1 的
圖片**——來源不變，產物「圖」就不該變動。例如：來源只修改了某個
物件框內的文字，整張圖就**只有對應處變化，其他部分照舊**。

**整理後解讀**：
- R9 的 ⚠ 張力正式解決：**方案 A（確定性程式 renderer）是目標**，
  方案 B（LLM 代轉）只是初期過渡（bootstrap 階段可接受，非終態）。
- 「語法制定得好 → 程式可轉」明確把**可機械渲染性**變成語法設計的
  驗收條件：語言不得包含只有 LLM 才能詮釋的模糊語意。
- 1:1 穩定的三個等級被一次講全（此為 R8 的最終強化版）：
  1. 同來源 → 同圖（確定性，位元級）
  2. 改框內文字 → **只有該處變**（局部性：修改的影響範圍與語意
     範圍一致，不得外溢）
  3. 寫作者是誰（人/AI/編輯器代寫）不影響結果（來源等價 → 圖等價）
- 「改框內文字只變該處」隱含排版約束：文字變長時框的尺寸策略
  （固定框裁切/換行？框微長但不推動全局？）需要在渲染標準裡明訂
  ——這正是 R8「更詳細的標準」的具體例子。→ 規則已定，見 R10。

### R10 — 框尺寸策略：依「有無明確尺寸宣告」分流（2026-07-02）

**原意**：「文字變長時框只微幅變大、不推動全局」這個例子應該**有條件**
——取決於這個「框」物件有沒有明確限定 width/height（數值或比例）。
若有，新增文字自然得**等比縮小放入**（字體可能略為縮小）。

**整理後解讀**：
- 這是 R8 優先權規則（explicit > auto）在尺寸維度的具體化，
  內容適應策略二分：
  1. **有明確 width/height（數值或比例）**：框尺寸神聖不可變——
     內容遷就框，文字等比縮小（字級微降）以放入。
  2. **無明確尺寸**：框遷就內容，微幅變大，但**不推動全局**
     （局部性守恆，R9.1 穩定性第 2 級）。
- 通用原則可提煉為：**明確宣告的屬性是剛性約束；未宣告的屬性彈性
  適應，且適應的外溢範圍要最小化**。渲染標準的每個屬性都應照此
  模板定義其「未指定時的適應行為」。
- 尺寸宣告支援**數值或比例**兩種形式（比例＝相對於什麼？畫布？
  父容器？——細節留到語法設計時定）。
- 字體縮小屬於引擎的自動適應手段，作者不管字級——與「作者寫 what、
  引擎管 how」的宣告式原則一致。

### R11 — 制定方法論：借鑑現成標準、寧少勿濫、通用規則優先（2026-07-02）

**原意**：這些標準的制定都應該**參考既有的現成標準**，且應該像一般
**協議制定**那樣：**寧可優先支援的少**，但不該開創過多特例。應以
「**通用**」規則優先考慮。

**整理後解讀**：三條制定方法論：
1. **借鑑優先（survey before invent）**：每個設計點先盤點現成標準
   （SVG、CSS、Mermaid、DOT/Graphviz、D2、ELK、OOXML…），能借語意
   /語法/慣例就借，自創是最後手段。與 usecases.md 的 build-vs-buy
   方法和 R7「貼近 AI 已熟慣例省 token」一脈相承。
2. **保守擴張（RFC 精神）**：初版功能面寧小勿大——支援少而穩定的
   核心，勝過大而特例叢生的表面積。未支援 = 明確拒絕（呼應 v1
   封閉文法），日後按需擴充；不為單一場景添一次性特例。
3. **通用規則 > 特例清單**：遇到新需求先問「既有通用規則能否涵蓋」
   （如 R10 的剛性/彈性模板、v2 的「為何不能是原語+樣式」審查），
   而不是加新指令。規則數量是語言複雜度的真正度量（也直接決定
   R7 的 prompt token 成本）。
- 這條與 v1/v2 的既有工程紀律完全同向：v2 草案的「四原語封死」、
  「新結構必須論證為何不能是原語+樣式」就是此方法論的先例。

### R12 — 首版範圍：量大內容優先 + 預留擴充彈性（2026-07-02）

**原意**：標準 draft 第一版應優先從**圖片最常出現的內容**開始支援
（量大的需求優先滿足）。初期能看多遠（未來可能要支援哪些新需求）也
可納入考慮——**這時候語法的擴充彈性就要保留**。

**整理後解讀**：
- 首版範圍由**頻率統計**決定，不由技術趣味決定：先滿足語料中
  出現量最大的圖型。與 R11「寧少」合成 = 少而準：砍到只剩高頻核心。
- **待辦（重要）**：需要對使用者的實際 Word 文件圖片做一次
  **圖型普查/分類統計**（哪些類型、各佔多少、各類的構成元素），
  作為首版範圍的依據。R0 的「約一半可被 mermaid 描述」是初估，
  正式定範圍前應有更細的盤點。已知樣本：refer-images/（網路拓撲、
  查表鏈、封包格式——usecases.md 場景 4/5 的 A/B/C 三類）。
- **擴充彈性 ≠ 擴充實作**：首版不做未來功能，但語法結構要為它們
  留位——具體手段（既有先例）：
  - 版本頭（v1 的版本頭模式）+ 相容性規則
  - 關鍵字開頭的行文法：新能力 = 新關鍵字，舊解析器可精準報錯
  - `key=value` 選項槽：新屬性不破壞位置參數
  - profile 機制（R7）：新領域 = 新 profile，核心不動
- 「看得多遠」的既有素材：usecases.md 場景 1–5 就是未來需求清單
  （協議動畫、演算法、協議庫、封包走訪+配置表、硬體查表鏈），
  可作為「擴充彈性是否足夠」的思想實驗測試集。

### R13 — 預設值 = 最大宗需求（最小補充原則）（2026-07-02）

**原意**：文字的通用性上，應優先把「**預設**」解釋為**最大宗需求**
——這樣文字需要「補充」（額外宣告）的機會相對低。

**整理後解讀**：
- **慣例優於設定（convention over configuration）**：每個屬性的
  預設行為對準統計上最常見的用法，讓多數文件「什麼都不補」就是
  對的。補充宣告只出現在偏離主流的少數情況。
- 效果是三贏，直接強化多條前案：
  - 文件更短 → AI 產出與讀取的 token 都省（R7）
  - 未宣告面積大 → 引擎自動適應的空間大（R8/R10 的彈性側）
  - 人類手寫負擔低、可讀性高（R3 的 MD 式易讀）
- 設計程序上的要求：定每個屬性的預設值時，必須問「**大多數圖在
  這裡長什麼樣**」，而非「什麼在實作上最方便」。預設值是統計
  決策——與 R12 的頻率普查同源（普查結果同時餵範圍與預設值）。
- 與 R10 模板合成後的完整屬性模型：**未宣告 → 用最大宗預設並
  彈性適應；已宣告 → 剛性遵守**。
- 推論：預設值本身也是標準的 normative 內容（不同實作必須有相同
  預設，否則同文件不同圖，違反 D1）。

### R14 — Bootstrap 策略：在任何工具支援之前如何可用（2026-07-02）

**原意**：待解決的關鍵問題之一：初期在任何 editor/reader/viewer 都不
支援的情況下，該如何滿足應用？使用者的想法：在被支援以前，先以
`XXX.fd` 儲存對應內容，同步生成 `XXX.svg` 嵌入 MD 檔案內。

**整理後解讀**：
- 提案是一個 **sidecar 慣例**：`XXX.fd`（真相來源）+ `XXX.svg`
  （生成產物）+ .md 以普通圖片引用嵌入 SVG。這在今天*所有* Markdown
  viewer 都能用（GitHub、VS Code、Obsidian……），完全不需要生態系
  配合——bootstrap 問題靠「讀取端不需要認識 FigDown」解決。
- **待解的弱點**：來源被移出 .md——AI 讀 .md 時只看到圖片連結，
  看不到文字（弱化 R9 的「單檔雙通道」模型）。候選緩解手段
  （可並用）：
  (a) **同名慣例**（normative）：`figure.svg` ⇔ 同目錄 `figure.fd`；
      教 AI agent「看到 X.svg 就去讀 X.fd」。
  (b) **自攜來源的 SVG**：renderer 把 FigDown 來源嵌進 SVG 本身
      （`<metadata>`/`<desc>`），並附來源 hash——產物自帶真相、
      可偵測過期（直接滿足 R9 的配對/失同步需求）。
  (c) .md 內嵌 ```figdown 圍欄塊（放在圖旁或取代圖），等 viewer
      原生渲染的那天——Mermaid 的終局模式。
- 建議分期：**Stage 0** = sidecar 慣例 + CLI 生成器（今天就全域
  可用）；**Stage 1** = 圍欄塊由插件渲染（VS Code 擴充、隨採用度
  成長爭取 GitHub 式原生支援）；sidecar 慣例永久保留，作為最低
  公分母的 fallback。
- `.fd` → `.svg` 的生成時機需要定義：手動 CLI、編輯器存檔時、
  file watcher、pre-commit hook、CI——可標準化為建議（非強制）
  工作流。

### R15 — 先行者盤點：相鄰專案存在，但目標交集無人做（2026-07-02）

**起因**：使用者問「目前沒有類似專案嗎？這需求似乎必然是趨勢」。
快速掃描結果（2026-07-02；完整 `prior-art.md` 深度盤點列為後續工作）：

- **D2**（Terrastruct）——最接近的鄰居。現代宣告式 diagram 語言、
  多排版引擎（dagre/ELK/自研 TALA）、確定性（同輸入→同輸出）、
  部分位置控制（`near`），定位就是「比 Mermaid 更好」。
- **Mermaid**——LLM 已經在寫的事實標準（GitHub 原生渲染），但排版
  控制極弱、位置不能釘。
- **PlantUML / blockdiag 家族 / WaveDrom / Graphviz**——合計圖型
  覆蓋廣（timing、nwdiag、bitfield）但分裂在多語言；硬體文件要拼裝
  好幾家。
- **draw.io / Excalidraw**——重要先例：`.drawio.svg` /
  `.excalidraw.svg` 把來源嵌進 SVG（= R14 的自攜來源產物）。但
  GUI-first，「來源」是座標序列化，對 AI 無語意可讀性。
- **學術線（DiagrammerGPT 等）**——LLM 規劃 layout 生成圖像；目標
  是生成，不是可維護的知識來源。

**FigDown 的差異化（無既有專案承諾）**：
1. 跨修改的**排版穩定性**作為規格保證（D1 第 2 級：局部改→局部變）
   ——D2 的確定性只覆蓋相同輸入；加一個節點仍可能全圖重排。
2. **釘定參數剛性、自動排版繞行**作為核心公理（D2 有部分能力，
   非規範性承諾）。
3. **來源↔產物的規範性配對**（SVG↔文字的同步/過期偵測入標準）。
4. **GUI = 文字編輯**的編輯器契約。
5. 硬體/網路文件圖型（bitfield+波形+拓撲+查表鏈）**單一語言**全覆蓋。
6. **AI 教學 token 成本**作為一級設計指標。

### R16 — Template：圖的「種類」即應用樣板（2026-07-02）

**原意**：回歸「template」概念——以我們普查分類出的圖種類為本。每種
圖想表達的資訊目標不同，所以**相同的 keyword 可以有不同的意義**。
應該把「種類」視為樣板（一種應用），這是先天優化的好處。

**整理後解讀**：
- 一套核心文法（詞法、屬性模型、pin/layer 語意），多個 **template**
  ＝ 按圖型分的詞彙表與預設值：
  1. 關鍵字按 template 釋義：`node` 在 `topology` 是設備（預設圖示
     風格）、在 `flowchart` 是步驟（矩形）、在 `block` 是模組——
     由 template 消歧。
  2. 預設值按 template 統計：R13 升級——每個 template 的預設值取自
     普查**該桶**的統計（flowchart 預設 `flow down`、block 預設
     `flow right`、topology 預設無向邊……）。
  3. AI 教學單元按 template 切（R7 最大化）：教核心 + 當下使用的
     template 即可。
- **普查的桶 = template 清單與優先順序**；classified/ 樣本資料夾
  直接成為各 template 的設計語料與驗收集。「分類即設計」。
- 順帶解決 node 形狀問題：`kind` 詞彙表按 template 定義；`shape=`
  保留為跨 template 的展現層覆蓋。
- 語法表面：版本頭宣告 template，如 `figdown 0.1 block`。型別區塊
  （bitfield/table/wave）本身是 template，也可嵌入場景文件
  （混合文件，見 OQ-S5）。
- 護欄（R11）：新 template 必須有語料佐證——不得增生。Mermaid
  驗證過此模式（首字宣告圖型），但 FigDown 的 template 共享同一
  核心文法，不是 N 個獨立語言。

### R17 — 表格：MD 心智模型 + 合併/顏色；從樣本庫盤點需求（2026-07-02）

**原意**：table 需要支援儲存格合併（rowspan/colspan）。Markdown 的
表格已經做得很好，我們的差異在於：(a) 把它「圖片化」，(b) 在 MD 的
基礎上擴充——合併、顏色、以及**從目前收集的圖片庫**盤點出的其他
特殊需求。

**整理後解讀**：
- 澄清：GFM 表格其實*不能*合併儲存格——這個缺口正是 FigDown 價值
  的一部分。MD 表格仍是心智模型（R11 借鑑），FigDown 補上合併 +
  儲存格級顏色 + 圖片產物。
- 語法已入草案 §4.2：未加引號的 `^` = 向上合併（rowspan）、`<` =
  向左合併（colspan），加引號為字面字元；`cell r,c color=…` 做
  儲存格級標註（標註附著於定址）。
- **方法（使用者指示）：從 `classified/table-matrix/`（212 張唯一
  樣本、加權 917）盤點表格特徵清單**——多層表頭、對齊、框線、
  欄寬……普查資料夾就是每個 template 的需求來源（R16「分類即
  設計」）。
- **盤點已完成（2026-07-02，5 個視覺 agents、212 張樣本）**：
  cellcolor 58.5%、merged 41.5%、headercol 41.0%、multiheader
  34.9%、multitable 20.8%、colwidth 20.3%、partialborder 17.0%、
  memmap 14.2%、rowhl 12.3%、symbol 11.8%、素表僅 6.6%、alignmix
  1.4%、rotated 0.5%。後果：multiheader 升為 must-have（草案 + PoC
  已加可重複的 `head` 行）；alignmix/rotated 依證據自 v0.1 砍除；
  partialborder/memmap 列 v0.2 候選。原始數據：
  figdown-samples/results/table-features.*.tsv。

### R18 — 語法跟隨主流，按採用率加權（2026-07-02）

**原意**：一條長期原則：語法上應盡可能 follow 相關標準的主流——
而「主流」必須**按受歡迎程度/使用率加權計算**來認定。

**整理後解讀**：
- 細化 R11 的「借鑑優先」：同一構造存在多個既有慣例時，不隨意挑——
  選採用度最大的那個（使用者數、repo 數、renderer 支援度、AI 訓練
  語料中的出現量）。受歡迎度要**量測**，不是憑感覺。
- **首要理由（使用者，2026-07-02）：抗幻覺。** LLM 寫不熟悉的語法
  時會往它熟悉的方向漂移——把自創構造「自動校正」成主流方言的
  寫法，幻覺出看似合理實則錯誤的行。與主流對齊 = 生成時順著模型
  先驗走而不是對抗它：語法幻覺更少，寫→驗→修迴圈收斂更快。
- 次要理由鏈：採用度 → 人類熟悉（零學習成本）→ AI 先驗知識
  （R7 的 token 節省）→ 工具互通。
- 每個語法決策的實務規則：列出前 2–3 個既有慣例、估計相對使用率、
  採用領先者——除非它與 FigDown 公理衝突（此時記錄理由）。
- 稽核備註（回溯套用）：用此規則覆查現有決定——如 edge 箭頭
  `->`/`--`/`<->`（Mermaid/D2/DOT 主流 ✓）、`#` 註解
  （shell/Python/YAML 主流 ✓）、表格 rowspan 標記：我們的 `^` vs
  markdown-it-multimd-table（採用度最高的 MD 表格合併擴充）用的
  `^^`——**待辦：查證使用率，若 `^^` 勝出則對齊。**

### R19 — 推廣靠有意義的範例藝廊（2026-07-02）

**原意**：要有效推廣 FigDown，需要大量 PoC，而且必須是*有意義的*
圖片範例：常見協議標頭、協議協商過程、經典演算法與資料結構演示。
這些需求都要滿足。

**整理後解讀**：範例藝廊既是行銷素材、也是 R14 的 dogfood：每個
範例是入庫的 `X.fd`+`X.svg` 成對檔案，藝廊 index 直接嵌入 SVG——
repo 本身就在示範「一份來源、兩種讀者」。規劃（四波 E1–E4、
里程碑 G1–G4）：[gallery-plan.zh-tw.md](gallery-plan.zh-tw.md)。
後果：`sequence` template 提前（受眾重心所在，儘管本語料僅
0.6%；語法依 R18 必須貼齊 Mermaid）；演算法演示先以靜態圖序列
起步（頁/幀模型，R1），待動態 profile 出貨再上動態。

### R20 — 標籤支援數學運算式語法（2026-07-02）

**原意**：需要支援常見的「數學相關運算式語法」。

**整理後解讀**：演算法/資料結構藝廊的標籤與儲存格需要
O(n log n)、Σ、下標、上標、分數。主流認定（R18）：`$…$` 包夾的
LaTeX 數學（KaTeX/MathJax 慣例；GitHub MD 已原生渲染）。待決設計
點：確定性零依賴的子集（上下標、希臘字母、運算子以 Unicode 對映）
vs. vendor KaTeX。先做子集，用 E3 範例量測實際需求。

### R21 — 單機價值：零採用也有用（2026-07-02）

**原意**：即使這個標準沒有被廣泛推廣，單就 editor → SVG（下載）、
同時保存 `.fd`、下次載入後微調修改就能產生新圖——這個迴圈本身
就已經很有價值。

**整理後解讀**：
- FigDown 的底線是一個**自足的作圖工具**：在 editor 裡撰寫、匯出
  在所有 Markdown viewer 都能用的 SVG、隨時回來微調。生態採用是
  上檔空間，不是前提——這為整個專案去險（對照 D2 依賴 viewer
  支援的困境，R15）。
- 因為產物內嵌自身來源（R14/D1），往返其實不需要 sidecar 紀律：
  **把 SVG 本身拖回 editor 就能還原來源**，SHA-256 同時驗證完整性
  （不符則警告「SVG 在生成後被改過」）。已在 PoC 實作（Load
  按鈕 + 拖放）。
- 推廣含義（R19）：「你的圖永遠不會變成死檔」是頭條賣點——每張
  匯出的圖永遠可再編輯，與手畫圖片形成根本對比。

### R22 — 語意保真，非像素保真（2026-07-02）

**原意**：重建既有圖片時不需要 100% 復刻——要說清楚的是原圖想表達
的*拓撲*與*元件之間的關係*。這就是 text 的本意：**實質在於
meaning；人類視覺閱讀是二維或三維，而文字描述只有一維。**

**整理後解讀**：
- 「FigDown 能不能表達圖 X」的驗收標準是**語意完備**（原圖的每個
  元件、關係、隨附知識都在、且無歧義），不是視覺復刻（圖示美術、
  精確曲線、裝飾性對話泡）。
- 這句話是專案本體論的一句話版本：知識以語意存在；文字是它的
  一維編碼（AI 讀的）；圖是它向人類知覺的確定性二維投影
  （R3 的「兩種讀者」在*維度*層次的重述）。
- 對藝廊（R19）的實務後果：廠商圖重建以「原圖說的東西是否全部
  被捕捉」評判——srl-evpn-irb 與 vxlan-encap 兩例不靠圖示字型、
  對話泡、大括號就已達標。

### R23 — RFC ASCII 圖是基本目標語料（2026-07-02）

**原意**：RFC 裡有大量 ASCII 圖範例；這些圖應列為我們基本的解決
目標。

**整理後解讀**：
- RFC ASCII art 是「figures as text」的*始祖*——自 1970 年代運行
  至今，證明工程師接受文字編碼的圖。FigDown 是它的自然繼承者：
  同樣的一維來源哲學（R22），加上確定性二維渲染、顏色、與機器
  可讀的語意。
- RFC 圖型清單對映到 templates：封包佈局 → `bitfield`（已完成，
  msb0）；訊息階梯/時序圖 → `sequence` template（E2 前置）；
  狀態機（如 RFC 9293 的 TCP）→ 核心場景 + 狀態類 kind 詞彙；
  拓撲 → `topology`。
- 藝廊後果（R19）：精選一批著名 RFC ASCII 圖作為**驗收集**——
  「這份清單裡的每張圖都能用 FigDown 表達、且更好」是明確可驗證
  的推廣主張，正中 IETF 圈受眾。

### R24 — 語意優先的註記：`trunk` 先例（2026-07-02）

**原意**：像 LAG 橢圓這類 topology 需求，是否在語意上註明 trunk
就能自動標上（繪製）這個橢圓？這才是原文語意的精髓。

**整理後解讀**：確認並已實作。手工釘位的註記橢圓（展現層 idiom）
被 `trunk <id> "<label>" A--B, A--C` 取代：作者說出意義、繪圖慣例
歸引擎、推導出的圈圈零座標跟隨節點。這把 nudge 先例推廣成常設
規則：**當一個反覆出現的視覺慣例承載領域概念時，給概念一個指令，
永遠不要求作者手畫慣例。** 應走同一條路的候選：編號步驟 callout
（封包走訪）、冗餘配對、廣播域。

### R25 — GUI 畫出的圖必須產生「結構優先」的文字（2026-07-02）

**原意**：目標——即使人類用 UI（editor）畫出來的圖，背後實際的
文字意義也必須能直接、清楚地表達「結構」以及圖中所有元件之間的
關係。這是必要的目的之一。

**整理後解讀**：
- editor 回寫的品質底線：畫圖絕不能產出「座標湯」。結構
  （node/edge/group/bundle）以可用的最高語意層級捕捉（R24）；
  座標只是佈局附錄。
- **文字分層**：editor 物化的 `pin`/`size` 行收進尾部的
  `# layout` 區——結構宣告在前、附錄在後，AI 讀者在讀到任何
  佈局行之前（或完全不讀）就取得完整意義。PoC 已實作。
- **語意完備不變量（規範性、可測試）**：刪除所有 `pin`/`size`
  行後，文件必須仍可解析、仍可渲染（自動排版）、且表達*一字
  不差*的結構與關係。已加入測試套件；藝廊 9 個範例全數通過。

### R26 — Template 候選：`partition`（資源/配額圖）（2026-07-02）

**原意**：例如 per-port、per-queue utilization 相關的圖，標準要能
清楚表達。（使用者提供的參考照片僅為舉例，刻意**不**提交進 repo。）

**整理後解讀**：
- 圖型類別：一份總資源被分割為多個標記區域，**比例本身承載語意**
  （共享 vs 保留配額、上限/門檻為標記線、每欄再細分）。緩衝/配額
  分割與記憶體映射同族——普查已量得 `memmap` 佔表格 14.2%，此
  類別有語料證據。
- 已驗證**今天就能表達**（釘位+尺寸的方塊，本地示範、未提交）——
  但那是座標工，而且**過度建模**：使用者指正，此圖的真實結構是
  **N 個並排的框 + 橫向標記「線」**——著色區只是門檻線以下的
  填色，不是獨立的框。依 R24，門檻線是一級公民（它*就是*硬體
  語意：hi/lo threshold、cap）；區域由線推導。修正後草圖：
  `partition <id> "<label>"`／`column <id> "<label>"`（×N）／
  `line <id> "<label>" at=<百分比> [color=] [fill=below|above]`
  ——兩個 `line` 指令就重現參考圖；人永遠不算像素、也永遠不畫
  區域框。
- 渲染備註（使用者提問，已解）：「四個框」vs「一個框+三條分隔
  線」是同一張圖——引擎渲染為一個外框 + N−1 條內部分隔線（真實
  圖正是如此）。語法維持單一拼法（R18：同義兩拼法助長幻覺）：
  有標籤的圖逐欄宣告 `column`；匿名均分另有語意不同的簡寫
  `columns <N>`（「無標籤均分」，不是替代拼法）。**使用者裁決
  （2026-07-02）：兩種形式都支援；對參考例這類每欄皆有身分的圖，
  逐欄具名形式優於簡寫**（欄名本身就是知識；R25）。
- 狀態：**已被取代（2026-07-03，R28）**——專屬 template 上線次日
  即拆除：此圖可分解為通用構件（group + 透明欄位節點 + 新的通用
  `line` 導引線指令）。藝廊範例已照此重建。

### R27 — 3D／等角圖：可行、屬 chart 家族、低優先（2026-07-02）

**原意**：這類 3D（45 度）圖有可能做嗎？（參考圖：per-port/
per-queue 計數值的 3D 長條圖 + 半透明門檻平面。照片未提交。）

**整理後解讀**：
- **技術上可行且可確定性**：等角投影是純 2D 數學（每根柱 = 三個
  平行四邊形面 + 固定遮擋順序）；SVG 不需要真 3D。不是障礙。
- **本體論（R22）**：此圖的實質是*數據*（數值矩陣 + 門檻），不是
  結構。.fd 形態應為資料塊 + 門檻宣告；「等角」只是渲染樣式參數。
  歸 `chart` template 家族——census 權重約 1%，故低優先（R12）。
- 門檻平面就是 R26 門檻線的 3D 版——同語意、不同投影。
- **今天就可用**：同樣的知識以熱圖表達（table + 儲存格上色、超標
  格標記）同樣清楚——甚至更可讀（無遮擋）；已本地驗證。
- 狀態：**應使用者再次提出，最小版已提前實作（2026-07-03）**：
  `plot <table-id> [level=]`——表格即數據、bars3d 為確定性等角
  投影、level 平面即門檻線語意的三維重用。

### R28 — 通用性關卡：通用構件能表達就不創新詞彙（2026-07-03）

**原意**：keyword 命名很重要、要通用；新樣板要謹慎——若通用的
圖片繪製構件就能「表達」，就不該額外創造；冷門的使用方式都是
額外的包袱。（另：line 被要求命名 id 不好——沒有東西引用它。）

**整理後解讀**：
- 常設關卡，銳化 R11/R16：任何新 template 或關鍵字之前，先把圖
  分解到既有構件；只有真正不可化約的餘項才能成為新表面——且必須
  以其**通用**意義命名，不以觸發它的例子命名。
- 立即套用：`partition` template（partition/column/columns/line 共
  4 個關鍵字）上線次日拆除。不可化約的餘項恰好只有一個概念——
  「橫跨容器的導引/門檻線」——即現在的通用 `line` 指令（水位線、
  hi/lo 門檻、圖表標記線皆可複用）。欄位 = 群組內的透明節點；
  配額圖的特殊詞彙成本歸零。
- 推論：**id 只為引用而存在**——沒有東西能引用的指令（導引線）
  不帶 id。
- **Template 門檻——最終表述（使用者，2026-07-03）：除非「語意」
  上「完全沒機會」以既有構件精確表達，才該考慮獨立出新樣板。**
  語意不可表達性決定「能不能開」；語料頻率（R16）只決定「何時
  做」（優先序）。單憑頻率永遠不構成開設理由——partition 事件即
  案例：語料證據存在（memmap 14.2%）但語意可被精確表達（group +
  透明節點 + line/fill），故不設 template。現存型別區塊（bitfield／
  table／wave）通過不可表達性檢驗：任何「方塊+連線」的組合都無法
  精確陳述位元位置、儲存格網格、訊號軌道。

### R29 — Scope 跟隨語意；範例從意圖出發（2026-07-03）

**原意**：語意上可能要先明確，讓 user/writer 自己決定怎麼表達。
參考範例的原意是「兩個 GLOBAL 門檻配置一次、四個 group 共同
參考」——逐欄表達就說錯了意思。舉例的源頭必須以「想表達的
意義」為出發點。

**整理後解讀**：
- `line`/`fill`（以及未來的註記）以附掛目標提供 scope：群組級 =
  全域語意、節點級 = 逐元素語意。語言兩者都提供；**寫作者挑選
  能陳述意圖的 scope**——renderer 畫出來可能一樣，但文字對 AI
  的讀法不同，而那個差異正是知識本身。
- 範例撰寫規則：先確認圖的原意，再選讀起來與之相符的構件
  （buffer-partition 範例已依此從逐欄 fill×4 修正為一筆群組級
  fill）。

### R30 — 頻率換簡潔（2026-07-03）

**原意**：我們參數的精神：「越常見、大宗」，寫的越少。

**整理後解讀**：R13（預設 = 大宗）從預設*值*推廣到語法*長度*——
一條帶梯度的設計法則：
- 零字元：統計上的預設（完全不用宣告）
- 一個短 token：常見的顯式情況（`fill 15%`）
- 一個長 token：少見的顯式情況（`fill 15-35%`）
- key=value：僅限選用的偏離（`dir=down`、`color=`）
- **永遠不讓**常例為罕見能力付語法稅（被退役的 `from=`/`to=`
  正是那種稅）。
既有實例經此法則驗證：緊湊 `field a:1, b:2`、`columns N`、
`numbering=lsb0` 預設、GFM 表格原樣貼上。每個新參數的審查規則：
按預期頻率排列其使用情境，再由短到長分配拼法。

### R34 — Edge 標籤：行內三位置（2026-07-10）

**原意**：`P1 -- P2` 的兩端點常常需要說明其意義或角色。箭頭形式
本身（`A -> B`、`A <-> B`、`A <- B`、`A -- B`）在文字上就帶著圖、
且有清楚語意——兩端點有某種連結。但有時命名了 `A -> B`，還想多
描述 A 的角色或意義，且製圖上希望放在靠近 A 的位置。提議：用
`[ ]` 表達標籤在不同位置，且一條線**最多三個位置有意義**：
`A [label_for_A] <-[label_for_line]-> [label_for_B] B`——這是最大
可能性用法。

**整理解讀**：這是 OQ-S7 的收斂方向。三位置模型（尾/中/頭）在
主流是共識——Graphviz 直接命名這三個位置、PlantUML 把端點標籤
行內拼出（`Class1 "1" *-- "many" Class2`）、Mermaid 把運算子繞著
線中標籤拆開（`A -- text --> B`）；沒有任何被調查語言提供超過
三個位置（調查：[prior-art.zh-tw.md](prior-art.zh-tw.md) §1）。
行內拼法同時直接服務 R22：標籤在文字裡出現的位置就是它在圖面
所占的位置。若採納，`label=`／`taillabel=`／`headlabel=` 於同版
退役（R28），附機械式遷移。

**已定案（2026-07-10）**：採納並以 migration 0.1-dev.9 落地。
中括號內容規則依使用者後續裁示：內部平衡的中括號原樣巢狀
（`[flags[3:0]]` 完整顯示 `flags[3:0]`）；不平衡括號／`\n`／
字面引號改用引號形式 `["..."]`、套用標準字串跳脫（Mermaid
「形狀內引號」先例）；空 `[]` 為行錯誤。`label=`／`taillabel=`
／`headlabel=` 退役；整個藝廊由一支實作 0.1-dev.9 規則的腳本
機械式遷移完成——證明該規則確實機械（R31）。使用者同時重申
OQ-S4 維持出局：現在保守；日後真有需求，R31 之下「新增」是
便宜的。

### R35 — 兩個箭頭方向都收：`<-` 加入 `->`（2026-07-10）

**原意**：兩種都可以的話，對人類表達習慣而言，若不難支援就都
支援比較方便——就像習慣打 `Left -- Right` 一樣，是先天的習慣
問題。若作者就是想表達 `Left <- Right`，難道硬要打
`Right -> Left`？

**整理解讀**：原則上採納。`A <- B` 語意上就是 `B -> A`，R28/R30
通常會拒絕第二種拼法——但作者的*陳述順序*本身是人類編碼語意的
一部分（正在談論哪一邊），且 D2 的運算子集恰為 `-- -> <- <->`
（先例）。已隨 R34 改版落地（migration 0.1-dev.9）——運算子
註冊表只改了一次，不是兩次。

### R42 — 渲染選項 `no-title`：caption 只活一次（2026-07-18）

**原始需求**：實際嵌入時，多數圖在文件裡本來就有自己的
caption/title（觀察自下游 PDF 轉寫管線——其 `.fd` 同時以
`title` 與 provenance 註解攜帶原 caption）。SVG 內再畫一次標題
就是重複——渲染時要有「不顯示」的選項。

**整理解讀**：標題**文字**是語意（`.fd` 對 agent 保持自我描述）；
**畫不畫**是展現，且是逐產物的選擇（同一 `.fd` 可能嵌在 caption
之下、也可能獨立檢視）。解法：**renderer 層選項**而非語言語法——
零新文法、註冊表不動、無需過 R28。`build-svg.js --no-title`、
lib `render(text,{title:false})`、markdown-it 外掛
`showTitle:false`。確定性協議：被接受的選項記錄於產物 metadata
（`data-render-options="no-title"`），產物維持（來源、記錄選項）
的純函數——規格 §7。編輯器 roundtrip 載入器容忍新增的 metadata
屬性。


**修正（2026-07-21，預設翻轉）**：維護者的設計法則審查抓到第一版
方向選反。依 R13，預設必須跟隨最大宗用法——而最大宗的產物是嵌在
宿主文件 caption 之下（R32 慣例、轉寫語料、本專案自身的 gallery），
主流工具亦不預設繪標題。故：**預設不繪標題**；`--with-title`／
`{title:true}`／`showTitle:true` 選擇加入（記錄為
`data-render-options="with-title"`）；編輯器屬獨立檢視情境，選擇
加入並記錄於其匯出。並重申公用標準的通則：旋鈕最小化——選項跟
語法一樣得證明必要性，其預設跟隨最高機率慣用，永不跟隨實作者的
第一直覺。

### R37 — 讀取語意也是規範：意義只由語法即可還原（2026-07-10）

**原意**：同時要確保 AI agent 引用我們的語法時，能清楚對應其意義

**整理解讀**：讀取方向獲得與撰寫方向同等的嚴謹度。原則：**構件
的語意必須不執行 renderer 就能從文字推導**；只在繪圖幾何中可見
的東西按定義即是展現，讀取端 agent **不得**由它推斷意義。本需求

範例抓到的「憑空填充」失敗類型。解法：規格為每個型別區塊加上
規範性的**「語意模型（讀取規則）」**段——bitfield（offset =
宣告寬度累加；任何地方都沒有隱含填充；`wrap` 純展現；
`numbering=` 只改標籤；`optional` → 條件 offset；`*` 之後為相對
offset）、table（邏輯網格＋合併；`colw`／顏色／對齊永不改變它）、
wave（tick = 軌道字元位置；跨 `gap` 連續）。AGENT-GUIDE §2 把
同一套規則教給讀取端 agent。

### D1 — Renderer 是確定性程式，不依賴 LLM（2026-07-02）

文字→圖的轉換由**純程式（無 LLM API）**完成，如 mermaid 模式；
LLM 代轉僅為初期過渡方案。語法設計必須保證可機械渲染
（無需 LLM 詮釋的模糊語意）。穩定性承諾三級：
(1) 同來源→同圖；(2) 局部修改→局部變化，其餘照舊；
(3) 人寫/AI 寫/編輯器代寫，等價來源→等價圖。

### D2 — 專案命名：**FigDown**（2026-07-02）

專案定名 **FigDown**（Figure + Markdown），tagline：
*"Figures as text — one source, two readers."*
- 定位一秒懂：「Markdown 的圖形層」；命名法與 Markdown 同家族。
- 副檔名候選：`.figdown` / `.fd`；圍欄語言標記：```figdown。
- 已查 GitHub 無重大撞名（2026-07-02）。落選者：Graphein（蛋白質
  圖庫佔用）、Isograph（GraphQL 框架佔用）、Semagram（學術專案
  佔用）、Duograph、Textagram（可用但棄選）。
- 計畫放上 GitHub 集思廣益、開放完善。

### D3 — 文件語言慣例：英文為主、繁中並行（2026-07-02）

為了在 GitHub 上集思廣益，文件以**英文為主檔**：`xxx.md`（英文）、
`xxx.zh-tw.md`（繁體中文版）並行維護。本筆記即依此慣例：
`requirements-notes.md`（EN）+ `requirements-notes.zh-tw.md`（本檔）。

### D4 — pin 採絕對 px；第一次拖曳凍結全圖（2026-07-02）

源自 PoC 實測（使用者拖一個節點、整張圖走樣）。三條規則，已入語法
草案（§3，OQ-S2 定案）：
1. `pin at=` 採**絕對畫布 px**——比例座標實測後否決（畫布一長大，
   所有比例釘點跟著漂移，違反 pin 存在的目的）。
2. **Pin-on-first-touch**：使用者第一次拖曳時，編輯器把*所有*節點
   位置物化成 `pin` 行；此後佈局完全手動、演算法退場。自動排版也
   一律計算全部節點（被釘的節點保留自己的格位），因此釘 X 永不
   使 Y 重排。
3. **edge 永遠是衍生物**——由節點邊框自動重算、自動適應，不可釘。

### D7 — 形狀歸幾何、語意歸標籤；`bundle`（2026-07-02）

兩項使用者裁決。(1) topology 詞彙**不得**綁定領域名詞（router/
gateway/host/switch……會無限增生）。節點只帶純幾何的 `shape=`
（box｜rounded｜circle｜ellipse｜cloud｜diamond｜cylinder，封閉），
裝置是什麼由標籤文字自己說（R22：語意在文字裡；R11：詞彙不得
增生）。`kind=` 退役並給出指路錯誤訊息。流程圖形狀併入同一軸
（decision→diamond、terminator→rounded、datastore→cylinder）。
(2) 鏈路捆綁指令 `trunk` 改名 **`bundle`**——中性統稱（LAG/ES/
port-channel/……）；`trunk` 在 VLAN 語境有歧義。

### D6 — 兩層釘點：群組成員採群組局部座標（2026-07-02）

使用者提案，與 v2 框架的兩層座標系一致：群組內節點的 `pin at=`
以群組局部原點（成員空間左上）為基準；群組本身由自己的 `pin` 以
畫布 px 錨定（未釘時原點由自動排版推導）。後果：拖動群組 = 修改
一行文字（成員免費跟隨）、文字 diff 獲得局部性、群組內編輯永不
干擾其他群組。編輯器已實作（pin-on-first-touch 現在寫入群組錨點
+ 群組局部的成員釘點）。

### D5 — 表格內容採 GFM 管線語法；合併跟隨 multimd-table（2026-07-02）

R18 的第一次完整應用。使用者選擇 `|` 分隔而非空白分詞；table 區塊
因此直接容納**原封的 GFM 管線列**（`|` 為註冊的行首 token——封閉
文法不破；現有 Markdown 表格貼上即用；LLM 生成 GFM 幾乎零幻覺）。
必要的 `|---|` 分隔列標出表頭邊界（其上多列 = 多層表頭），其冒號
給出各欄對齊。核心 GFM 沒有 spans，合併因此跟隨採用度最高的 MD
擴充 **markdown-it-multimd-table**：`||` = 向左 colspan、單獨的
`^^` 儲存格 = 向上 rowspan。GFM 之外的 FigDown 能力維持關鍵字行
（`cell r,c color=…`、`cell r highlight`）。本決議廢止 R17 暫定的
`^`/`<` 標記，並結案 R18 對表格的稽核項。

**已考慮並否決（2026-07-02）：以空格位置表達對齊。**
提案：`|text |` = 靠左、`| text|` = 靠右、`| text |` = 置中。
R18 稽核：沒有任何開放標準這麼做（GFM 明訂 trim 儲存格空白；
AsciiDoc/Textile/Org 的對齊都用顯式記號）。否決理由：(1) 破壞貼上
相容——Prettier 等格式化器任意補空白，貼上的表格會得到隨機對齊，
跑一次 formatter 就默默改圖（違反 D1 穩定性）；(2) 隱形字元載語意
是經典失敗模式（Makefile 的 tab），也是 LLM 的幻覺源。每欄對齊
維持 GFM 冒號；若日後真需要每格對齊（普查僅 1.4%），將採
`cell r,c align=left|right|center`。

### D10 — 專案以持久化名維護者對外發聲（2026-07-16）

伴隨 [PROCESS](../spec/PROCESS.zh-tw.md) 的治理裁決：社群互動
（issue、release note、文章、對外推廣）由一個持久的化名維護者
身分承擔，而非無面孔的 org 官方口吻、也非真名。理由：標準的
採用者需要一個一致的「人」的存在來累積信任（「誰在裁決？找得到
人嗎？」）；實質內容由證據鏈流程保證；且化名**單向可逆**——
日後可升級連結真實身分，反之不可能。org 帳號仍是名義發布者；
維護者 handle 是聲音。

### D9 — 語意類：`class` ＋ 衍生 legend（2026-07-10）

OQ-S8 依證據定案。語料實測
（[prior-art.zh-tw.md](prior-art.zh-tw.md) §4.4）顯示顯式 legend
很少（加權 ≈3%），但 **56% 的圖以顏色/線型承載分類語意而無對映
陳述**——FigDown 轉寫會遺失的暗語意（R37 盲區的語料尺度現形）。
已採納（規格 §2.7、migration 0.1-dev.10、純新增）：
`class <id> "<意義>" [展現預設]`，元素以 `class=<id>` 加入
（node/group/edge/field/cell），legend 條按宣告順序自動衍生。
元素顯式屬性覆蓋類預設（R8）。規範立場：顏色用於分類時作者
**應**用 `class`；裸 `color=` 是裝飾。Prior art：命名機制承
Mermaid `classDef`；沒有任何被調查語言把意義機器可讀地綁到元素
類上——這是 FigDown 的差異化而非為發明而發明（R18/R28 雙過：
56% 頻率＋今日語意確實不可表達）。

### D8 — Template 僅提供預設值（2026-07-10）

Review 結論（R33），收窄 R16：`figdown 0.1 <template>` 標頭保留，
但 template 只選定**預設值與驗證 profile**——預設流向、預設邊的
有向性、預設單位，各自對準該圖型的普查統計。template **不得**
改變標準指令的核心語意（`node` 永遠是節點、`edge` 永遠是關係）：
語意隨 template 而變，會迫使每個讀者攜帶逐 template 的語意，破壞
R28 所守護的單一核心模型經濟性。新 template 仍須同時具備語料佐證
**與**語意不可表達性（R28）。標頭同時攜帶版本相容規則：未知主
版本拒絕、未知次版本寬鬆解析、未知 template 在嚴格模式下拒絕。
