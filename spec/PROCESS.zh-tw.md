# FigDown 規格如何變更

> 可信標準的另一半（與測試套件並列）：可預期的變更流程。本文件
> 對專案自身的工作流具規範性。
>
> English version: [PROCESS.md](PROCESS.md)

## 1. 決策模型

FigDown 採**維護者裁決＋強制證據鏈**：

- 每項語言變更由維護者裁決。
- 沒有任何裁決能只憑品味成立：每一項都必須引用證據——語料統計
  （[census](../design/census.zh-tw.md) 或針對性量測）、prior-art
  調查（[prior-art](../design/prior-art.zh-tw.md)，採用度加權）、
  或可複現的實戰回饋。
- 每項裁決永久記錄為
  [requirements-notes](../design/requirements-notes.zh-tw.md) 的
  R（需求）或 D（決議）條目，**含被否決者與其理由**。決策日誌是
  專案的記憶；沒有檯面下的決定。

## 2. 新語法的關卡

新的指令、選項、列舉值或 template 進入語言，必須**同時**滿足
（R28/R18/R30）：

1. **語意不可表達**——該語意完全沒機會以既有構件精確表達。
   「比較方便」不構成理由；組合優於新詞彙。
2. **語料證據**——真實圖形需要它、且有量測到的頻率。頻率決定
   *何時*（以及拼法能多短——大宗寫最短）；不可表達性決定*是否*。
3. **Prior art 已調查**——先查主流圖語言、按採用度加權；能借
   就借（防幻覺：AI 作者已認得主流拼法）。

註冊表（規格 §10）是封閉的：未註冊即錯誤，新增只能走這道關卡。

## 3. 每次變更的義務

改動解析或渲染語意的變更，必須在同一 commit 內附上：

- **[MIGRATIONS](MIGRATIONS.zh-tw.md) 條目**：機械式改寫規則
  （非機械步驟明確標注）——vA→vZ 升級＝依序套用條目；
- 更新的 **[conformance](../conformance/README.md) goldens**——
  golden 只隨 migration 條目或偏差解決而動，永不默默改變；
- **雙語規格同步**（英文為規範文本、繁體中文平行）；
- 受影響的 agent 介面同步（AGENT-GUIDE、skill 包）。

## 4. 偏差必須大聲

實作與規格不一致時，偏差記錄於
[conformance/DISCREPANCIES.md](../conformance/DISCREPANCIES.md)，
並經深思熟慮地解決——引擎向規格靠攏、或修正規格，各留紙本軌跡。
把引擎行為默默凍結成事實規格，正是本流程要防止的失敗模式。

## 5. 版本

- 草案期：日期遞增的 dev 版號（`0.1-dev.N`），凍結時 squash 為
  `figdown 0.1`；migration 紀錄自該點重新起算。
- 凍結後：同一主版本內的變更必須向後相容；實作**必須**拒絕未知
  主版本；移除/改名需要主版本躍升＋migration 規則。
- 棄用：退役的拼法先成為**指名其 migration 條目的錯誤**（見已
  退役的 `label=` 家族），修法一查即得。

## 6. 提案管道

以 **syntax proposal** 模板開 GitHub issue。模板問的正是關卡所需：
你無法表達的語意、既有構件為何無法精確承載、真實（去識別）樣本、
你所知的 prior art。帶著證據來的提案得到裁決最快；「攻擊公理」
（告訴我們它們在哪裡會破）明確歡迎。
