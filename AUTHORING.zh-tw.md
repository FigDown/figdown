# 撰寫 SOP — 從「我需要一張圖」到正確的 FigDown

> 對象：**人類**（與 agent）在決定如何表達一張圖時參考。與
> [AGENT-GUIDE.zh-tw.md](AGENT-GUIDE.zh-tw.md)（agent 工作流）、
> [spec/PROCESS.zh-tw.md](spec/PROCESS.zh-tw.md)（語言如何變更）
> 互為配套。
>
> English version: [AUTHORING.md](AUTHORING.md)

## 第 1 步 — 先陳述意義，再選任何東西

寫出一句話：「*這張圖在陳述……*」。寫不出來，代表圖還沒準備好畫。
FigDown 語意優先：文字是 AI 讀為真相的東西，圖只是投影。以下每一
步都從圖的**意義**出發，永不從「它該長什麼樣」出發。

## 第 2 步 — 選 template（只是選預設值，不是選牢籠）

| 圖在講…… | Template |
|---|---|
| 封包標頭／暫存器的位元位置 | `bitfield` |
| 列 × 欄的數值（配置、狀態、映射） | `table` |
| 訊號隨時間 tick 變化 | `wave` |
| 裝置與其間的鏈路 | `topology` |
| 步驟、判斷、控制流 | `flowchart`（預設 `flow down`） |
| 元件、包含、資料流——其他一切方塊加連線 | `block` |

記住（D8）：template 只挑**預設值**（流向、單位）。每個指令在
每個 template 下都有效，選「錯」最多多寫一行顯式 `flow`，永不
限制你能說什麼。一個 `.fd` 也可以混用區塊：拓撲後面接它的 VLAN
`table`，是同一份文件。

## 第 3 步 — 用既有構件表達意義

想要新語法之前，先走完這張清單；
[pattern 庫](examples/patterns/index.zh-tw.md) 對多數列有現成骨架。

| 你想說…… | 用 |
|---|---|
| 這些元素屬於類別 X（以顏色/虛線呈現） | `class x "意義" color=… style=…` ＋ `class=x`——legend 自動衍生 |
| 框內的門檻／水位／區帶 | `line`（標記）與 `fill`（色帶）；掛在擁有該語意的 scope（R29） |
| 這些鏈路構成一個邏輯捆綁（LAG/ES…） | `bundle` |
| 線兩端的角色/埠/基數、線上的動詞 | 行內邊標籤：`a [1] -[places]-> [N] b` |
| 空間上屬於一起 | `group` ＋ `in=`；同列/同欄 → `rank` |
| 連向外界／來自外界（開放端箭頭） | `boundary ext "標籤"` ＋ 一條連到它的 edge——永不用假節點 |
| 貼著某節點的註記 | 小型 `style=dashed` 節點＋點線 edge（鄰近有語意時）或集中一張 `table`（密集資料時） |
| 同段位元依模式重新解釋 | 欄位上 `note="valid when …"`（OQ-S9） |
| 一張原圖裡有兩個概念區 | 拆成一概念一 `.fd`，由 Markdown 並排 |
| 表格儲存格內的趨勢 | Unicode 方塊 `▁▃▅▇` 作為儲存格文字（OQ-S12） |

表達時的兩條鐵律：

- **意義放文字、幾何歸引擎。** 永不把知識只編碼在顏色或位置裡；
  顏色用於分類時，必須掛在 `class` 上。
- **邊寫邊驗**：`node tools/build-svg.js X.fd`——錯誤是
  `Line N: message`；文法封閉，build 乾淨＝文件被完整理解。

## 第 4 步 — 都不合用？擴充提案指引

1. **先確認是「意義」缺口，不是「方便」缺口。** 檢驗法：用既有
   構件盡力表達，然後剝除所有展現（`pin`/`size`/`color`）——
   真的遺失了知識，還是只少了好看？「要多打幾行」不是缺口
   （組合優於新詞彙，R28）。
2. **查未決問題清單**——規格 §9（OQ-S…）追蹤已知缺口與其現行
   認可解。你的需求若在列，先用認可解，並把你的樣本補進該討論：
   語料證據正是推動 OQ 過關卡的燃料。
3. **永不因缺口卡住。** 先用純文字把缺的意義說出來（標籤、
   `note=`、`#` 註解標記位置），知識零損失地等語言跟上——
   且永不發明語法：未知行報錯是刻意設計。
4. **開 syntax-proposal issue**（模板即
   [PROCESS](spec/PROCESS.zh-tw.md) 關卡的鏡像）：你無法表達的
   意義・你的最佳嘗試與確切損失・真實去識別樣本與粗略頻率・
   你所知的 prior art。帶證據的提案得到裁決最快。
5. **新 *template* 請求門檻最高**——template 只是預設值包
   （D8），你缺的若是構件，請提構件。真正的新 template 需要
   語料證據**且**既有構件完全無法承載的語意（R28）。

## 第 5 步 — 你的提案會經歷什麼

依 [spec/PROCESS.zh-tw.md](spec/PROCESS.zh-tw.md)：分流 →
prior-art／語料調查 → 維護者附證據鏈裁決，永久記錄於決策日誌
（含否決）。被採納的語法與 migration 條目、conformance fixtures
同批落地——你的文件永不默默壞掉。
